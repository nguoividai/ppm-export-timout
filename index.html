Executive Summary
The Problem

Customers exporting PPM schedules for large contracts (≥2,000 visits) received a timeout error after 30 seconds. The export never completed. NG Bailey (Punit Segani, Senior Systems Analyst) reported the issue across multiple contracts — it was systemic, not isolated.

The Fix

The root cause was 5 separate correlated subqueries against the Job2Plnt_Task table — one per visit row. We replaced them with a single pre-fetched lookup, cutting database round-trips by ~80% for large datasets. All P0 acceptance criteria now pass.

Technical Deep-Dive

What Changed in the Code

GetPPMContractScheduleMsgHandler.cs · GetScheduleAsync

The export query builds a visit schedule by iterating over every PPMVisitSchedule row in the contract. For each row, it previously fired 5 separate LINQ-to-SQL subqueries against Job2Plnt_Task to count task states. The fix pre-fetches the needed columns once, then computes everything in memory.

Before — 5× subqueries per visit row
// Inside foreach visit loop — each .Count() hits the DB

var serviceTasks = a.JobAssetTask
    .Where(t => t.ServiceType
        == ServiceType.JobAssetTask)
    .ToList();                   // subquery 1

var total = a.JobAssetTask
    .Count();                       // subquery 2

var complete = a.JobAssetTask
    .Count(t => t.TaskStatusId
        == TaskStatus.Completed);      // subquery 3

var na = a.JobAssetTask
    .Count(t => t.TaskStatusId
        == TaskStatus.NA);             // subquery 4

var noSvc = a.JobAssetTask
    .Where(t => t.ServiceType
        == ServiceType.JobAssetTask)
    .Count();                       // subquery 5 (duplicate)
After — 1× pre-fetch, in-memory derivation
// Outside the loop — single projection, 2 columns only
var taskData = a.JobAssetTask
    .Select(t => new JobAssetTaskData {
        ServiceType  = t.ServiceType,
        TaskStatusId = t.TaskStatusId
    })
    .ToList();                   // 1 query total

// In-memory — zero additional DB round-trips
var serviceTasks = taskData
    .Where(t => t.ServiceType
        == ServiceType.JobAssetTask)
    .ToList();                   // in memory

var total    = taskData.Count();  // in memory
var complete = taskData
    .Count(t => t.TaskStatusId
        == TaskStatus.Completed);      // in memory
var na = taskData
    .Count(t => t.TaskStatusId
        == TaskStatus.NA);             // in memory
Why this caused timeouts

A contract with 5,700 visits issued 5,700 × 5 = 28,500 subquery executions in a single export request. Each hit the Job2Plnt_Task table individually. Entity Framework's lazy-loading translated each .Count() and .Where().ToList() into a separate SQL round-trip. The cumulative execution time reliably exceeded the 30-second command timeout before all rows were processed.
